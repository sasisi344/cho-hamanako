# 🖋️ Writing Style & Markdown Guide

## 🎭 役割とペルソナ

### 【絶対基準】熟練アングラー「私（さしし）」の一人称スタイル

この記事の語り手は、浜名湖を愛し、長年通い詰めている地元アングラー**「私（さしし）」**です。

- **一人称**: 必ず「私」または「私（さしし）」を使用してください。
- **トーン**: 基本は「です・ます」調ですが、単なる情報の要約ではなく、**「現場で見てきた経験」**を感じさせる重みを持たせてください。
- **視点**: 「初心者の頃の自分に教える」ような温かさと、「上級者も唸る現場のリアル」を両立させます。

### 読者ペルソナとトーン・カラー（執筆前に決める）

語り手は常に「私（さしし）」だが、**読者の釣りスキル習熟度**に応じて、ボリューム・専門性・語彙・説明の密度を変える。**まず「主な読者」を1つ決め**、それに合わせて記事タイプ（カテゴリ）を [.agents](../) の各 `*-article/SKILL.md` と照合する。

| 主な読者 | 用語・語彙 | 説明の立ち位置 | 距離感・印象 |
|----------|------------|----------------|--------------|
| **初級** | 専門用語は**極力使わない**。どうしても出す語は初出で短く言い換え or 括弧補足。 | 手順・見分け方・「まずこれだけ」優先。比喩も可。 | 優しく伴走。上から説かない。 |
| **中級** | 用語は**使ってよい**。初出で一言定義すればよい。 | 現場のコツと**理由**（なぜそうするか）まで書き、説明力で納得させる。 | 同じ現場のベテランが隣で話す距離感。「こちらが一段深い知識を持っている」ことを**説明の質**で示す（煽らない）。 |
| **上級・分析（深掘り・データ重視）** | 用語を**積極的に**使い、必要なら括弧・一文で定義。 | **アナリスト視点**: 観察・仮説・根拠・結論。一般論より「この水域・この条件での整理」。ややアカデミック寄りの段落構成（序論→観測→考察→示唆）でよい。 | 査読に耐える密度。ただし地元の身体知は必ず接続し、冷たい教科書にならないようにする。 |

**カテゴリとの対応（目安・固定ではない）**

- **初級寄りになりやすい**: ガイド系、ターゲットの入門寄り、旅行・グルメなど「初来訪・初挑戦」の文脈。
- **中級が主戦場になりやすい**: ポイント記事（ピラー）、季節・月次レポートの実践パート。
- **上級・分析寄りになりやすい**: 深掘り（`fukabori`）、習性・潮流・地形を因果でつなぐ記事。各タイプの SKILL にある**最低文字数・セクション比率**はそちらを優先。

**本文のボリューム（サイト共通の目安）**

- **通常記事**（guide / season / travel / theory 等）: 本文は**2,000文字以上を必須下限**とする。理想は 2,500〜3,500 字。モジュール構成は [base-article-structure/SKILL.md](../base-article-structure/SKILL.md) を参照。
- **ポイント単体ピラー**（`category: points` の1ポイント記事）: **3,000文字以上**（[point-article/SKILL.md](../point-article/SKILL.md) 優先）。
- **まとめ・複数ポイント紹介**: 全体 2,000字以上。各ポイントは差別化中心（[point-comparison-matrix.md](../point-article/point-comparison-matrix.md)）。
- 意図的に短くする場合（速報・補足ノート等）は、チーム内で理由を共有できるよう冒頭か要約で明示する。
- 深掘りなど、タイプ別 SKILL で**より大きな最低文字数**がある場合は、**タイプ別を優先**する。

## ✒️ 文章構成とボリューム（方向性）

### ✅ 現場至上主義（Reproducibility）

- 「場所による」といった曖昧な表現を避け、「潮がこの向きに動いた時は〜」「この杭の周りは〜」といった、具体的で再現性のある解説を優先してください。
- 読者がその記事を読んで、**「明日現場で何をすべきか」**が明確になるボリューム（1セクションあたり300〜500文字程度）を維持し、**全体として**上記の文字数目安・タイプ別要件を満たす。

### ✅ 記事の「体温」

- 冒頭と結びには、必ず「私（さしし）」の最近の釣行の感想や、読者への熱いメッセージを一言添えて、血の通った文章にしてください。
- 専門用語の扱いは**主な読者ペルソナ**に合わせる（初級では言い換え優先、中級以上で現場用語を効果的に使用）。上級・分析トーンの記事でも、結びは読者の行動やマナーに接続し、エンゲージメントを残す。

## 🛠️ Markdown 記法ルール

### ✅ 強調 (Emphasis)

- **Production Rules**: 本番用（`src/content/`）では、強調に必ず `<strong>強調内容</strong>` タグを使用してください。`**` は使用禁止です。
- **SEO Strategy**: 1段落あたり1箇所程度の自然な密度を維持し、重要キーワードを強調します。

### ✅ 1文1段落ルール (Mobile First)

- **1 Sentence = 1 Paragraph**: 文章のリズムを作り、モバイルでの圧迫感を減らすため、**1つの文が終わるごとに必ず空行を1行挟んでください**。

### ✅ スペースの扱い

- 日本語と英数字・記号の間には **スペースを入れない**。
- 例：`AstroSphereで構築する9月のレポート。`

### ✅ 編集メモ `> [!forAI]`（remark-github-alerts）

**目的**: 記事をリライト・追記するときに、本文へ盛り込みたい論点・別視点・事実確認メモを**短い箇条書き**で置く。人間とエージェントが「次に何を本文へ反映するか」を共有するための**内部向け合図**とする。

**書き方（推奨）**:

- 記事末尾に **1ブロック** にまとめる。
- 本文のドラフト全文は書かず、**指示・メモ**に留める（例: 親子それぞれの視点、季節リスク、周辺店舗名、マナー注意の追加など）。

**エージェント・執筆者の義務**:

- 該当ファイルを編集するときは `> [!forAI]` の内容を読み、可能な範囲で**適切な見出し・段落・既存の CAUTION/TIP 等**へ取り込む。サイト内の他記事（ポイント記事など）と店名・距離が食い違う場合は**実在ページを優先**して整合させる。
- **`draft: false` で公開する前**に、`> [!forAI]` ブロックは**必ず削除**する。ビルド後も remark アラートとして**読者画面に表示されうる**ため、下書き専用のメモを本番記事に残さない。

**補足**: `[!NOTE]` `[!TIP]` `[!CAUTION]` 等は読者向け。`[!forAI]` だけが「編集用メモ」という意味合いのラベルとして使う。

## 🛠️ コンポーネント使用ガイドライン

### ✅ 内部リンク (BlogCard)

- 形式： `<BlogCard slug="slug" />`
- 記事冒頭に `import BlogCard from "@components/BlogCard.astro";` が必要です。

**slug の引き方（厳守）**:

- `slug` prop にはリンク先記事のフロントマターに定義された **`slug:` フィールドの値**をそのまま渡す。ファイルパスではない。
- 例: `src/content/blog/target/seabass/tactics/index.mdx` に `slug: seabass-tactics` とあれば → `<BlogCard slug="seabass-tactics" />`
- **【捏造禁止】**: `.workspace/` 配下のインデックスファイルまたは実在するファイルの frontmatter を必ず確認すること。推測で書かない。

**URL の自動解決ルール**（BlogCard 内部ロジック）:

- `category: points` の記事 → `/points/{slug}`
- それ以外の記事 → `/blog/{slug}`

### ✅ 道具紹介 (TackleCard)

- 形式： `<TackleCard id="category/id" />`
- **【重要: IDの捏造禁止】**: `src/content/affiliates/` 配下にある実際のディレクトリ名とID（ファイル名）を組み合わせて指定してください。
- 記事冒頭に `import TackleCard from "@components/TackleCard.astro";` が必要です。
